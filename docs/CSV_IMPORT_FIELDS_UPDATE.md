# CSV Import Fields Update

## What Changed

Updated the CSV import to better organize product information with clearer separation between description, features, and metadata.

## Key Changes

### 1. **Description is Now Optional** ‚úÖ

- The `description` field is now truly optional
- Can be left empty if product features tell the whole story
- Use for general product summary when needed

### 2. **Details = Product Features** ‚ú®

The `details` field is specifically for customer-facing product features (bullet points):

**Example:**

```csv
details: "with FREE Kolin KCF. 07TMAGAC,Full DC Inverter for 60% Energy Savings,Multi-Stage Air Filtration"
```

This matches what customers see in product listings:

- ‚úì with FREE Kolin KCF. 07TMAGAC
- ‚úì Full DC Inverter for 60% Energy Savings
- ‚úì Multi-Stage Air Filtration

### 3. **New Metadata Field** üÜï

Added dedicated `metadata` field for structured product information:

**Example:**

```csv
metadata: "{""brand"":""Kolin"",""model"":""K4G-100WCINV"",""power"":""1.0 HP"",""type"":""Window Type""}"
```

Use for:

- Brand
- Model
- Product type
- Power/capacity
- Color
- Size
- Any structured attributes

### 4. **Specifications Field** üìã

Keep using `specifications` for additional technical specs:

**Example:**

```csv
specifications: "{""warranty"":""1 year"",""energyRating"":""5 star"",""voltageRange"":""220-240V""}"
```

## Updated CSV Structure

### CSV Format

```csv
sku,name,description,category,vendor,retailPrice,sellingPrice,costPrice,images,details,metadata,specifications
011700162641,Kolin K4G-100WCINV 1.0 HP,,electronics,TECH-001,25495,22995,20000,"url1.jpg,url2.jpg","Feature 1,Feature 2,Feature 3","{""brand"":""Kolin""}","{""warranty"":""1 year""}"
```

### Database Result

```json
{
	"sku": "011700162641",
	"name": "Kolin K4G-100WCINV 1.0 HP",
	"description": null,
	"specifications": {
		"details": ["Feature 1", "Feature 2", "Feature 3"],
		"metadata": {
			"brand": "Kolin"
		},
		"warranty": "1 year"
	}
}
```

## Field Usage Guide

| Field            | Purpose          | Format          | Required | Example                     |
| ---------------- | ---------------- | --------------- | -------- | --------------------------- |
| `sku`            | Product SKU      | String          | ‚úÖ Yes   | `011700162641`              |
| `name`           | Product name     | String          | ‚úÖ Yes   | `Kolin K4G-100WCINV 1.0 HP` |
| `description`    | General summary  | String          | ‚ùå No    | `Latest 2024 model`         |
| `category`       | Category slug    | String          | ‚úÖ Yes   | `electronics`               |
| `vendor`         | Vendor code      | String          | ‚úÖ Yes   | `TECH-001`                  |
| `details`        | Product features | Comma-separated | ‚ùå No    | `Feature 1,Feature 2`       |
| `metadata`       | Structured info  | JSON            | ‚ùå No    | `{"brand":"Kolin"}`         |
| `specifications` | Tech specs       | JSON            | ‚ùå No    | `{"warranty":"1 year"}`     |

## Benefits

### For Users

- ‚úÖ **Clearer structure** - Know exactly where to put each type of information
- ‚úÖ **Optional description** - Don't need to duplicate features
- ‚úÖ **Better organization** - Features, metadata, and specs are separated

### For Frontend

- ‚úÖ **Easy to display** - Features are in an array, ready for bullet points
- ‚úÖ **Easy to filter** - Metadata is structured for queries
- ‚úÖ **Better UX** - Clear separation of information types

### For Database

- ‚úÖ **Structured data** - Metadata is queryable
- ‚úÖ **Consistent format** - All products follow same structure
- ‚úÖ **Flexible** - Can add more fields without breaking structure

## Migration Notes

If you have existing CSVs, update them to use the new structure:


```csv

metadata: {"brand":"Kolin","model":"K4G-100WCINV"}
specifications: {"warranty":"1 year"}
```
